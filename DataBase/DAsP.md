# 데이터아키텍처 준전문가
# 1. 전사 아키텍처 이해
### 1.1 EA(전사 아키텍처)
- EA(Enterprise Architecture)
  - EA 도입은 IT 혁신/관리 및 통제/시스템의 도입, 구축, 운영, 평가 등을 통합적으로 관리
  - **일반적 목적은 IT 투자 대비 효과를 최대화 하는 것**
- 전사(Enterprise)
  - 전사는 매우 유동적인 개념!!
  - 한 개의 기업에서 여러 전사가 나올 수 있음
  - 전사는 다수의 사업 영역으로 구성
- 아키텍처(Architecture) 구성 요소
  - 규칙(Rule)
  - 계획(Plan)
  - 모델(Model)
### 1.2 EA 프레임워크
> EA 프레임워크는 기업의 전략과 아키텍처, 인프라를 통합한 모델
- ![Image](https://github.com/user-attachments/assets/4dbf9834-90a5-4dfc-af3f-7459260b2ce7)
- EA 정책(수립 방향)
  - 아키텍처 매트릭스
    - 조직 내 다양한 계층의 사람들이, 매트릭스에 포함된 산출물이 범위와 목적에 적합하게 정의되었음을 확신할 수 있어야 함
    - 이 또한 **유동적!!**
      - 하단의 아키텍처 도메인도 일반적인 분류인 거지, 고정이 아님
    - 뷰(아키텍처 정보 유형)(아키텍처 도메인)
      - 비즈니스
        - **BA를 우선적으로 정의한다**
          - 기술 참조모델과 헷갈리지 말 것
      - 애플리케이션
      - 데이터
        - 개념 Data Model은 Entity만
      - 기술
      - 각 도메인은 상호 연관성을 갖는다
    - 관점(의사결정 유형)
      - 계획자
      - 책임자
      - 설계자
      - 개발자
    - 아키텍처 매트릭스도 유동적
  - EA 비전
  - EA 원칙
- EA 정보
  - EA 정보는 변하지 않는 구성요소를 대상으로 함
    - EA 정보 구성요소 여러개를 통해 EA 산출물이 탄생함
      - EA 정보 구성요소는 서로 중복되지 않으며, 상호관계가 유기적으로 연결됨
  - **EA 정보 구축은 아키텍처 매트릭스에서 정의한대로 진행**
  - 현행 아키텍처
    - 상위 분류체계를 정의(우선적) -> 하위 정의는 병렬적으로 진행
  - 목표 아키텍처
    - EA 도입 초기에는 상위 수준까지만 진행
  - 이행 계획
    - 현행 -> 목표 아키텍처로 전환하기 위한 단계별 계획
- EA 관리
  - EA 관리 체계
    - EA 관리 체계(ex 제도화)는 구축 후 바로 진행!
    - EA 관리 조직
      - EA 정보 전담 조직이 없다면, EA는 하나의 문서에 불과할 뿐임
        - 대기업의 경우에는 정보 전담 조직이 있어야겠지
    - EA 관리 프로세스
    - EA 관리 인력
  - EA 관리 시스템
    - 정보 정의 도구 영역
      - EA 정보 생산 및 Modeling 도구(도식화)
    - 정보 관리 영역
      - 아키텍처 정보 저장/관리 Repository
    - 정보 활용 영역
      - 정보 검색/활용하는 포털 제공
  - EA 평가
- 대표 프레임워크
  - 자크만(Zachman)
    - 5관점, 6묘사방법
  - 미연방정부(FEAF)
    - 참조 모델 기반
  - 국방성(DoDAF)
    - 무기 체계
  - 미국재무성(TEAF), 한국전산원
    - 4관점 아키텍처 매트릭스 제공
  - ToGAF(민간 표준)
    - 각종 참조 모델 활용 관계 정의
### 1.3 EA 참조모델
- 구축 방법
  - 공통 특성 추출 / 산업군에 맞게 범용적으로 만들기
    - 요소 간 경계나 하위 수준의 정의가 모호할 수 있음
  - 대표 기업 선정 -> 표본으로 삼아 재활용
    - 기업의 보안상, 대부분 공개되지 않음
- 사례
  - 성과 참조 모델
    - 성과에 대한 기여도 평가
  - 업무 참조 모델
  - 서비스 참조 모델
    - 서비스 요소 분류를 위한 기능 중심의 참조 모델
    - 응용 서비스 재사용 촉진 + 중복 개발 방지
  - 데이터 참조 모델
    - 데이터 상호 운용성/교환 촉진 + 중복 배제/재사용
    - DBMS는 Data 참조 모델보단, 기술 참조 Model + 포준 프로파일
  - 기술 참조 모델
    - 기술 표준/규격 정리 모델
    - 인프라 표준화 + 벤더 독립성 향상
    - 시스템 간 상호 운용성, 이식성, 확장성 향상
    - 몇 개의 Service 영역으로 구성
    - **활용도가 가장 높음**
      - 가장 먼저 적용
### 1.4 EA 프로세스
- EA Process는 IT 관리체계 프로세스에 포함되는 개념
- EA Process는 EA Framework의 구성 요소
- EA 비전 수립
- EA 구축
  - EA 환경 분석
  - EA 구축 방향 정의
  - EA 프레임워크 정의
- EA 관리
- EA 활용
  - 목표 아키텍처 이행 계획
    - 현재 상태 -> 목표로 하는 아키텍처 전환을 위한 실행 계획
  - EA 정보 상시 활용
    - IT 기획 관리
      - ex) 업무 프로세스 혁신, 정보화 계획 수립
      - ex) 중복을 배제한 효과적인 시스템 투자계획 수립에 활용한다.
    - IT 구축 관리
      - ex) 개발, 프로젝트 계획
      - ex) 기존 시스템 개선 및 신규시스템 개발을 위해 기준 및 참조정보를 제공한다 
    - IT 운영 및 통제
      - ex) 도입되는 시스템이 전사 표준을 준수하는지 통제함으로써 상호운용성, 유지보수 편리성 등을 확보한다. 
      - ex) 시스템 변경 시에 영향도를 파악하여 위험을 최소화한다.
### 1.5 EA VS DA
- 통합성
  - 범위 통합성
    - **EA 모델 내** 불일치성 제거
      - 다양한 구성 요소들이 범위와 목표를 공유하도록 함
  - 수평 통합성
    - **도메인** 불일치성 제거
      - 동일 계층 내의 일관성 유지
  - 수직 통합성
    - **관련 관점** 불일치성 제거
      - 다양한 계층 간의 일관성을 유지

## 2. 데이터 요건 분석
### 2.1 생명 주기 도형
- 정보 요구 사항 수집
  - 사용자의 필요와 기대 파악
- 정보 요구 사항 분석 / 정의
  - 분석 -> 요구사항 정의
- 정보 요구 사항 상세화
  - 요구 사항 세부적 구체화
- 정보 요구 사항 검증(검토/리뷰)
  - 요구 사항이 정확하고 일관되게 정의되었는가?
### 2.2 정보 요구 사항 유형
- 외부 인터페이스 요건
  - 시스템 간 발생하는 모든 입출력에 관한 요건
    - 데이터 송수신 방식 추가 or 변경
  - 관리 기준
    - 중복성
      - 동일 형태의 인터페이스가 중복 정의되지 않도록
    - 표준 준수도
      - 인터페이스 관련 국제 표준 및 국가 표준 존재 시
- 기능 개선 요건
  - 출력을 생성하는 주요 활동 및 프로세스에 대한 개선 요건
  - 관리 기준
    - 불가변성
      - 재변경되지 않도록 근본적인 개선 방안 마련
    - 범용성
      - 많은 사용자가 편리하게 사용할 수 있는 요건 우선 고려
- 성능 개선 요건
  - 관리 기준
    - 실현 가능성
      - 현행 기술 수준과 서비스 특성 고려, 구현 가능 여부 확인
    - 측정 가능성
      - 구체적이고 측정 가능한 형태로 제시
- 보안 개선 요건
  - 관리 기준
    - 실현 가능성
      - 구현 가능 여부 확인
    - 불가변성
      - 근본적 개선 방안 마련
### 2.3 정보 요구 사항 조사
- 수집
  - 관련 문서
    - 경영 계획에 대한 자료
      - 기업 비전, 목표, 전략 이해
    - 정보 시스템에 대한 자료
      - 시스템 구조, 기능, 성능 파악을 위해
    - 과거 컨설팅 보고서
      - 현재 문제점 + 개선 방향 파악을 위해
    - 전산 처리 업무 메뉴얼
      - 업무 처리 절차 + 시스템 사용 방법 이해를 위해
    - 협업 부서 업무 자료
      - 협업 현황과 업무 연계 이해를 위해
  - 사용자 면담
    - 계획 및 준비
    - 면담 수행
    - 면담 결과 분석
    - 분석 결과 피드백
    - 2~3인 구성
    - 면담자, 기록자, 관찰자
      - 기록자는 사전지식 + 주요 내용 확인
      - 관찰자는 **주의 환기** + 보충 질문 + 최종 판단
      - 면담 대상자는 업무에 대해 명확히 설명할 수 있는 사람으로 선택
  - 워크샵
    - 이해당사자들의 득실 차이가 많을 때 효과적
    - 목적
      - 경영층/부서장 공통 의견 도출
      - 면담 비용 절감
      - 전문가 판단력으로 최적 결론 도출
      - 정보 System 관리 수준, 문제점, 현안 파악(전산 전담 부서 면담 수행 목적)
  - 현행 업무 조사서
  - 현행 프로그램 데이터
- 정리
  - 우선 순위 분석
    - 화폐 가치 산출 방법
      - 경제적 가치 평가 -> 우선순위 결정
      - 정보 요구사항 간 관련성 -> 1~5점
      - System/기업 차원 중요성 -> 1~3점
    - 상대적 중요도 산정 방법
      - 조직 목표에 미치는 영향 평가 -> 우선순위 결정
      - 업무 기여 수준에 따라 1~5점
        - 매트릭스 작성 -> 가중치 -> 가중 평균 -> 중요도 산출 -> 우선 순위 부여
- 통합
  - 정보 요구 사항 목록 검토
  - 정보 요구 사항 목록 통합 / 분할
- 질문법 세션 진행 시
  - Tie-down 질문법
    - 동의 유도(~라 생각하시죠?)
  - 대안 진보
    - 여러 선택지 中 1개 선택
  - 포커핀 or 부메랑
    - 질문에 질문으로
  - nominal group
    - 개인 아이디어 수집 -> 집단 논의
    - 정보 수집법의 일종, 적합X
- 조사 기법
  - Activity 분석기법
    - 기업차원의 data model 도출
    - RAEW
      - Role, Activity, Entity, Worker 관계 분석
### 2.4 정보 요구 사항 분석
- 분석 대상 정의
  - 현행 업무 분석 대상 정의
    - 현행 업무 흐름도, 업무 설명서, 업무 분장 기술서 등 수집으로 업무 영역 파악
    - 분석 대상 업무 영역 선정 -> 업무 기능 분해도 작성 -> 업무 체계적 분류
  - 현행 시스템 분석 대상 정의
    - 현행 시스템 구성도, 설계 및 개발 보고서, 테이블 정의서, 유지보수 이력 등 수집 -> 시스템 구조 + 기능 이해
    - 시스템 유용성, 완전성, 정확성, 유효성 평가
- 정보 요구 사항 상세화
  - 프로세스 관점
    - 수행 절차
      - 프로세스 중심으로 계층도, 정의서 작성
        - 업무 조사서를 기반으로 함
        - 프로세스 목록과 업무 흐름도를 포함함
      - 기본 프로세스 기준으로 정보 항목 정리
      - 표준화 과정 통해 통합성 / 분리성 여부 검토
        - 최종적 정보 항목 목록 정의
    - 수행 작업 내용
      - 프로세스 분해 / 상세화
        - 하향식으로 프로세스 분해 -> 프로세스 계층도와 정의서 작성
        - 일반적으로 3차 수준까지 분해
          - 분해 수준은 업무 특성을 고려하여 결정
      - 정보 항목 도출 및 표준화
        - CRUD 기능 구분하여 기술 -> 정보 항목 도출
        - 정보 항목은 명명 규칙을 준수하여 명명
          - 업무 용어를 그대로 사용, 명사형으로 기술
      - 정보 항목별 통합성, 분리성 여부 검토
        - 프로세스별 정보 항목 분류
        - 동음이의어, 이음동의어 여부 파악 -> 통합 or 분리 검토
    - 수행 작업 지침
      - 프로세스 분해 깊이
        - 업무 특성을 고려해 분해 수준 결정, 일반적으로 3차 수준
      - 프로세스 명칭
        - 내용을 파악할 수 있도록 함축적이고 유일한 이름 부여
      - 프로세스 계층도 작성
        - 프로세스 도출을 위해 계층도 작성 
  - 객체 지향 관점
    - 유즈케이스 다이어그램
      - Actor, System이 제공해야 하는 기능 Usecase
    - 유즈케이스 상세화
      - 도출된 Usecase의 사건 흐름을 구조화
      - 정상적 흐름 선 기술 -> 예외 사항 후 기술
    - 클래스 다이어그램
      - 엔티티 클래스 도출 
      - 관계, 클래스 도출
        - 관계 식별 및 명명
      - 속성 정의
- 정보 요구 사항 확인
  - 재검토 계획 수립
    - 검토 기준 정의
      - 기준: 완전성, 정확성, 일관성, 안정성

### 2.5 정보 요구 사항 검증
- 주제별 분류
  - 요구 사항 분석가 수행
    - 요구 사항을 직접 수집하고 분석한 담당자가 자체적 검토 수행
    - 수집된 요구사항의 논리적 오류, 중복, 누락 확인
    - 자체 분석에 의한 객관성 저하의 문제점 발생 가능
    - 관련 업무 팀과의 의사소통 원할 -> 추가인력 투입 없이 원할히 진행
    - 업무에 대한 이해도가 높으므로, 상관분석을 통한 정확한 업무 분석 가능성 높음
  - 품질 보증팀 수행
    - 독립된 품질 보증팀이 요구 사항 품질을 객관적으로 평가
    - 표준과 절차를 준수하였는가? 명확하고 검증 가능하게 작성되었는가?
    - 업무에 대한 이해도는 낮으나, 상관분석작업을 통해 업무 이해도를 높일 수 있으며, 전체적 인터페이스 검증에 용이
    - 낮은 업무 이해도로 인해, 일부 사안에 대한 정확한 분석을 통해 단점을 지적하여 수정하기 어려움
  - 외부 감리 수행
    - 외부 전문가나 기관이 독립적 시각에서 요구 사항 검토
    - 업무 파악에 한계가 있으나 제 3자의 시각으로 검토 가능
    - 내부 인력이 효과적으로 지원하지 않을 경우, 상황에 맞지 않는 분석 결과가 나올 수 있음
    - 상관분석의 객관성을 극대화할 수 있음
- 정보 요구 / 애플리케이션 상관 분석
  - 사용자와 이해관계자의 요구 사항을 수집하고 문서화
  - 애플리케이션 기능 상세 나열
  - 상관 매트릭스 작성: 매트릭스 형태로 배열 -> 각 요구 사항과 기능 연관 표현
- 정보 요구 / 업무 기능 상관 분석  
  - 업무 프로세스와 기능 정의 및 문서화
  - 업무 기능이 필요로 하는 정보 요구 사항 식별 및 연결
  - 상관 분석 수행
- 정보 요구/애플리케이션 상관 분석에서 특이 상황 정리
  - 생성X 사용처X
    - 사용처가 없기 때문에 정보 불필요 도출 / 프로세스 누락 확인
  - 생성O 사용처X
    - **참조하는 신규 프로세스 도출** 요망
  - 생성X 사용처O
    - 기본 프로세스 도출
    - 삭제 여부 검토
    - 해당 업무영역으로 이동
